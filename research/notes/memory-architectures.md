# Memory architectures — distilled notes

Source: Phase-0 memory-research agent over the 5 papers (`research/papers/`) + Claude review.
Design home: `foundation/memory-and-ledger.md`. Fact mirror: `memory/facts/0004`.

## Synthesis

LoopRR needs three things the literature already solves in pieces: durable cross-session/cross-repo shared memory, a temporal+spatial "clock-in ledger," and deterministic Record-and-Replay. None of the five papers were built for a multi-agent software team across GitHub repos, but each contributes one load-bearing primitive. The opinionated synthesis: borrow MIRIX's typed-memory routing as the schema, but collapse its six human-centric types into four dev-centric ones (Core/persona, Procedural/playbooks, Semantic/codebase-and-decision facts, Episodic/event-ledger), and drop the Knowledge Vault into your existing secrets manager rather than reinventing it. Borrow MAGMA's dual-stream write path (fast synchronous append, async LLM consolidation) and its temporal+entity+causal edge model — but reject MAGMA's four-orthogonal-graph storage as over-engineered for v1; you do not need a causal graph engine to ship. Borrow GAM's JIT/deep-research insight wholesale: keep memory LIGHT and lossless-by-reference, never pre-compress away the raw artifact, because in software the ground truth (the commit, the diff, the CI log, the PR) already exists in a queryable store (git/GitHub) — this is GAM's "page-store" handed to you for free. Borrow Mem0's production discipline: ADD/UPDATE/DELETE/NOOP fact reconciliation, soft-delete-for-temporal-reasoning, and the hard latency numbers that justify "don't put a synchronous graph build on the critical path." Borrow SemaClaw's harness architecture as the actual implementation spine: three-tier context (working/external/persona), plain-Markdown user-owned memory, hooks as the clock-in mechanism, and DAG Teams' two-phase plan-then-execute as the natural Record half of Record-and-Replay.

The storage decision is the highest-leverage call and the papers strongly bias it. Do NOT make a graph DB or a vector DB your system of record. Your system of record is Postgres + git/GitHub. The clock-in ledger is append-only event data with time, duration, and a virtual-location tag — that is a textbook Postgres table (timestamptz, tstzrange/duration, repo+env+branch+worktree as the spatial key, worker_id, status, event payload jsonb), optionally with TimescaleDB if event volume ever justifies it; it is emphatically NOT a vector problem and NOT a graph problem. MAGMA's own ablation shows the temporal backbone is an immutable ordered chain — in your world that chain IS the ledger and the git commit DAG. Put a pgvector index on the semantic/episodic text columns for hybrid retrieval (Mem0/SemaClaw both land on BM25+dense hybrid with keyword as a first-class signal, which matters enormously for code where identifiers must exact-match), and reserve any graph store for a later phase if cross-repo causal "why did build X break" queries become a real product need. Entity/causal relations can live as foreign keys and a `relations` join table in Postgres long before they need Neo4j.

Record-and-Replay is where you must be most careful and where the papers help least, because none of them do deterministic execution capture — they do memory, not replay. "Replay" in LoopRR should mean two distinct things kept separate: (1) Coordination replay — a structured, queryable transcript of what a routine did (the DAG plan it generated, tool calls, observations, diffs, decisions), which GAM's page-store + SemaClaw's session-export JSON + Mem0's structured facts give you, and which is genuinely useful for refining future work; and (2) Deterministic replay in the strict systems sense (same inputs → same outputs), which LLM nondeterminism makes impossible to guarantee and which you should NOT promise. Capture the inputs (prompt, injected memory snapshot, tool results, model+params, RNG seed where available, git SHAs) so a run is reconstructable and auditable, and let "replay" mean deterministic re-execution of the captured DAG plan (SemaClaw's deterministic scheduler running an LLM-authored graph) — not deterministic regeneration of the LLM's tokens. This is the honest, shippable version.

Two cross-cutting risks dominate. First, async consolidation correctness: MAGMA and Mem0 both lean on an LLM to infer causal/entity links and to resolve ADD/UPDATE/DELETE conflicts, and both papers flag that extraction errors propagate silently. In a software team this is worse than in chat — a wrong "fact" about which API is current, or a mis-consolidated decision, will be acted on by a worker that writes code. Gate consolidation behind conservative thresholds, keep raw provenance (the SHA/PR/log it came from) on every fact so any claim is verifiable, and prefer soft-delete + supersession over destructive UPDATE so the ledger stays auditable. Second, multi-tenant / multi-repo isolation (SemaClaw's explicit concern): shared memory across repos is a feature AND a leakage vector — scope every memory row by repo/org and namespace per-worker, and treat cross-repo reads as an explicit, permissioned join rather than a global pool, mirroring your existing governed-write-boundary instinct.

## Adopt (idea — source — how in LoopRR)

- **Typed memory with a routing manager (six types: Core/Episodic/Semantic/Procedural/Resource/Vault), each with its own schema and update manager, plus a meta-router that decides which type(s) a new event touches.** — *MIRIX*. Collapse to FOUR dev-native types as the canonical schema: Core (per-worker persona + standing instructions, akin to CLAUDE.md/SOUL.md), Procedural (reusable playbooks: how-to-release, how-to-open-PR, how-to-handle-CI-failure), Semantic (durable facts about codebases, APIs, architecture decisions, owners), Episodic (the clock-in ledger itself). Map MIRIX 'Resource' to a pointer into git/GitHub (don't copy artifacts); map 'Knowledge Vault' to your existing secrets manager, NOT a memory table. A lightweight router (single LLM call) tags each incoming event with target type(s); store as a `memory_type` enum column, not separate databases.
- **Dual-stream write path: a synchronous fast path that only appends the event and updates the immutable temporal backbone + vector index (no blocking LLM call), and an asynchronous slow path worker that dequeues recent events and does LLM consolidation (causal/entity linking, summarization).** — *MAGMA (mirrored by Mem0's async summary and confirmed by Mem0's Zep critique)*. Clock-in/status-post = synchronous INSERT into the ledger table + enqueue (e.g. Postgres LISTEN/NOTIFY, pg-boss, or a Supabase edge function trigger). A background consolidation worker batches recent ledger rows, extracts Semantic facts and cross-repo relations, and writes them back with provenance. Compaction/consolidation must never block the worker's critical path.
- **JIT memory + lossless page-store + a 'researcher' that plans→searches→integrates→reflects at query time, keeping stored memory light and treating the complete history as the searchable substrate.** — *GAM*. Memory stores compact memos + stable references (repo, SHA, PR#, file:line, run-id). When a worker needs context, a 'researcher' step issues hybrid retrieval over memos, then dereferences to pull the exact diff/log/issue via the GitHub MCP tools (get_commit, get_file_contents, get_job_logs, pull_request_read) — this is GAM's page-fetch. Adopt GAM's three retrieval tools verbatim: vector, BM25/keyword, and direct-ID lookup (the ID case = fetch by SHA/PR#). Cap reflection depth (GAM default 3) to bound cost.
- **LLM-driven fact reconciliation with explicit ADD/UPDATE/DELETE/NOOP operations against semantically-similar existing memories; mark contradicted facts invalid rather than physically deleting them to preserve temporal reasoning.** — *Mem0*. On consolidation, retrieve top-k similar facts, let the LLM pick ADD/UPDATE/DELETE/NOOP. Implement DELETE as a `valid_until` timestamp + `superseded_by` FK, never a row drop. Every fact row carries provenance (source SHA/PR/run) so it's verifiable, not just asserted.
- **Three-tier context architecture (working memory + compaction, external retrieval memory with 3-level degradation Vector+FTS→FTS→keyword-scan, persona partition) stored as plain user-owned Markdown files; hooks as runtime lifecycle clamp points; DAG Teams two-phase orchestration (LLM emits an explicit dependency graph, deterministic scheduler executes it).** — *SemaClaw*. (a) Clock-in = a SessionStart/task-start hook that writes a ledger row (worker, time, repo/env/branch/worktree = virtual location); status updates = tool-post hooks; clock-out = task-complete/error hook recording duration+outcome. (b) Persona tier = per-worker SOUL.md/CLAUDE.md committed to the repo. (c) External memory = Postgres+pgvector with SemaClaw's exact degradation fallback so retrieval still works when embeddings are down — index docs without stopword filtering, filter queries, to favor recall@index/precision@query. (d) Record-and-Replay: have the orchestrator emit an explicit DAG plan (inspectable, diffable, stored), then a deterministic scheduler replays THAT plan — gives failure-locality and reproducibility of coordination without pretending LLM tokens are deterministic.
- **Per-worker, append-only episodic ledger with event_type/summary/details/actor/timestamp fields, used as a structured calendar the agent can reason over.** — *MIRIX (Episodic Memory) + MAGMA (immutable temporal backbone)*. Postgres table: id, worker_id, event_type (clock_in|status|clock_out|blocked|handoff), summary, details jsonb, actor, repo, env, branch, worktree, started_at timestamptz, duration interval, ref (SHA/PR/run-id). This single table gives every worker and the human temporal+spatial awareness via plain SQL/views; surface it as a 'team calendar' read model and feed it into worker context on session start.

## Reject

- **MAGMA's four separate orthogonal graph stores (semantic/temporal/causal/entity) with policy-guided beam-search traversal as the v1 retrieval substrate.** — Over-engineered for launch and the wrong cost/benefit for software. MAGMA's own ablation shows causal and temporal carry most of the value while entity-only scores worst; and in LoopRR the temporal 'graph' is just the ledger/commit-DAG and entity links are FKs. Standing up four graph views + an LLM causal-inference consolidator is a large eng+latency cost (and MAGMA lists storage/complexity as an explicit limitation). Start with Postgres relations; promote to a graph DB only if cross-repo causal queries become a proven product need.
- **MIRIX's Knowledge Vault as a memory component for secrets/credentials/API keys.** — Storing credentials in the memory system violates least-privilege and creates a leak surface. Secrets belong in your existing secrets manager / env, referenced by handle. Keep them out of the memory substrate entirely.
- **MIRIX's multimodal screenshot pipeline and the 'Agent Memory Marketplace' (tokenized/decentralized memory trading).** — Zero relevance to a code-team product and a distraction. LoopRR's modalities are text/code/logs/diffs, all already structured. The marketplace vision is speculative and orthogonal.
- **Promising deterministic, bit-for-bit replay of agent runs ('the RR' as true record-and-replay of execution).** — LLM outputs are nondeterministic across model/version/sampling, and tool environments (CI, network, GitHub state) drift. No paper here delivers deterministic LLM replay. Promising it sets a guarantee you can't keep. Scope 'replay' to (a) reconstructable/auditable runs via captured inputs+SHAs and (b) deterministic re-execution of the captured DAG plan.
- **Zep-style cache-a-full-summary-at-every-node temporal knowledge graph.** — Mem0 measured this at >600k tokens/conversation with hours-long async build and stale immediate reads. For a system that must give workers fresh shared state quickly, that footprint and latency are disqualifying. The lighter memo+reference model (GAM/Mem0) is strictly better here.
- **Pre-compressing artifacts (diffs, logs, PR bodies) into lossy natural-language memory at ingestion time (the AOT pattern).** — GAM's experiments show AOT compression causes irrecoverable information loss, and software ground-truth is exactly the kind of fine-grained detail (exact error line, exact arg) that lossy summaries destroy. Keep raw artifacts by reference in git/GitHub; summarize only as a retrieval index, never as the system of record.

## Risks

- Async consolidation can write wrong/hallucinated facts (MAGMA and MAGMA's limitations + Mem0 both flag LLM extraction error propagation). In a code team a bad fact gets acted on and written into source. Mitigate: conservative thresholds, mandatory provenance (SHA/PR/run) on every fact, soft-delete/supersession, and a human-visible 'facts pending review' surface for high-impact claims.
- Cross-repo shared memory is also a data-leakage / blast-radius vector (SemaClaw's multi-tenant isolation concern). A worker on repo A must not silently read private context from repo B. Mitigate: namespace every memory row by org/repo, make cross-repo reads explicit+permissioned joins, never a global pool.
- Latency creep if any consolidation/graph step sneaks onto the clock-in critical path (the exact failure Mem0 attributes to Zep). Mitigate: enforce that the fast path is INSERT+enqueue only; everything else is background.
- Compaction corrupts working state silently (SemaClaw: compaction errors are silent, compound, unrecoverable; their own prompt is code-biased but still imperfect). Mitigate: keep the lossless reference store so any compaction can be re-derived; re-inject standing constraints (CLAUDE.md/persona, open todos) across every compaction boundary.
- Clock/timezone and duration integrity in the ledger: workers across repos/CI runners may report skewed clocks. Mitigate: server-stamp timestamps at the write boundary (don't trust worker-supplied time), store timestamptz in UTC, derive duration from server-side start/end events.
- Ledger and memory write amplification at scale: every tool call posting a status row can explode volume. Mitigate: coalesce status events, sample, and consider TimescaleDB/partitioning only when volume proves it — don't pre-optimize.
- Vector index staleness right after a write (SemaClaw's 'mark day-log dirty on compaction' problem): a worker may not find what another just wrote. Mitigate: trigger incremental re-index on write, and always include a keyword/FTS fallback path so fresh exact-match content is findable before embeddings catch up.
- 'Replay' semantics confusion between stakeholders: engineers may expect deterministic re-execution. Mitigate: name the two modes explicitly in the product (Audit/Reconstruct vs Plan-Replay) and document the non-guarantee.
